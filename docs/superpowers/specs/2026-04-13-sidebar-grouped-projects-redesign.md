# GhostTerm 左侧面板整体整改设计

## 背景

当前左侧面板的核心交互仍围绕“最近项目下拉”构建：

- 顶部 `ProjectSelector` 只负责显示当前项目与最近项目下拉。
- 面板主体仍是 `Files / Changes / Worktrees` 三个标签页。
- 项目组织方式是“最近项目列表”，而不是“分组后的项目列表”。

这会带来几个问题：

- 顶部视觉结构过于工具化，不像一个真正的项目导航中心。
- “最近项目”只能按时间使用，无法按业务语义组织。
- 交互重心在一个简单下拉里，难以承载后续项目管理能力。

本次整改的目标是把左侧面板的项目组织模型从“最近项目”为中心，切换为“分组后的项目列表”为中心，同时只改左侧面板，不扩散到编辑器和终端区域。

## 目标

- 左侧面板顶部重构为“当前分组栏”而不是“当前项目栏”。
- 点击顶部下拉按钮后，展示“分组切换面板”。
- 左侧面板的主数据结构以“分组后的项目列表”为中心。
- 项目分组支持自定义，用户可以长期维护自己的项目分类。
- 搜索作用于当前分组下的项目列表，而不是最近项目全量下拉。
- 与现有后端最近项目列表兼容，不破坏当前项目切换链路。

## 非目标

- 不整改编辑器和终端布局。
- 不引入拖拽排序、多选批量移动、云同步。
- 不修改后端 `projects.json` 数据结构。
- 不实现高级搜索语法，只保留视觉样式与基础模糊搜索。
- 不做头像上传与复杂品牌定制。

## 设计原则

- 以“分组后的项目列表”为中心，而不是“最近项目列表”为中心。
- 后端继续维护项目打开历史，前端单独维护分组视图和分组映射。
- 视觉和交互优先贴近你给的参考图，但只实现必要能力，不额外扩展。
- 左侧面板继续是独立滚动容器，页面本身不滚动。

## 信息架构

整改后的左侧面板结构分为 4 层：

1. 顶部当前分组栏
2. 分组切换下拉面板
3. 搜索栏
4. 分组过滤后的项目列表

原有的 `Files / Changes / Worktrees` 标签页结构不在本次设计中删除，但左侧面板的首要交互重心从“文件视图”切换为“项目导航中心”。本次 spec 只定义顶部与项目区整改，不额外设计文件树内部结构。

## 分组模型

### 分组类型

- `全部`：虚拟分组，固定存在，不可编辑，不持久化为用户自定义分组实体
- `未分组`：真实保底分组，承接所有尚未被用户归类的项目
- `自定义分组`：用户手动创建的分组

### 项目归属规则

- 每个项目只属于一个分组
- 当前历史项目如果没有显式分组映射，则自动归入 `未分组`
- 删除某个自定义分组时，该组下所有项目自动移动到 `未分组`

### 计数规则

- 顶部分组计数显示当前分组下项目数量
- `全部` 计数为全部项目总数
- 每个分组在下拉面板中显示自己的项目数

## 数据模型

新增前端侧边栏分组状态，独立于后端最近项目列表。

### Group

- `id: string`
- `name: string`
- `emoji: string`
- `color: string`
- `createdAt: number`

### ProjectGroupingState

- `selectedGroupId: string`
- `groups: Group[]`
- `projectGroupMap: Record<string, string>`
- `searchQuery: string`

说明：

- `projectGroupMap` 的 key 使用项目路径 `project.path`
- `selectedGroupId` 支持 `all`、`ungrouped` 或某个自定义分组 id
- 分组状态仅前端持久化，不写入后端 `projects.json`

## 顶部当前分组栏

顶部栏不再显示“当前项目名称 + 路径”，而显示当前分组摘要。

### 组成元素

- 左侧：分组 emoji
- 左中：分组颜色圆点
- 中间：分组名称
- 右侧：项目数量胶囊
- 数量右侧：下拉箭头按钮
- 最右：分组编辑按钮

### 行为

- 点击下拉箭头：展开或收起分组切换面板
- 点击编辑按钮：打开当前分组编辑入口
- 当选中 `全部` 时，编辑按钮禁用
- 当选中 `未分组` 时，允许查看但不允许删除

## 分组切换下拉面板

参考你给的图，面板样式为悬浮卡片，显示：

- `全部`
- 所有自定义分组
- 当前项右侧勾选
- 底部分隔线
- `新建分组`

### 交互规则

- 点击某个分组：切换当前分组并关闭面板
- 点击 `新建分组`：进入创建分组流程
- 每个自定义分组行显示：
  - emoji
  - 颜色圆点
  - 分组名称
  - 项目数
  - 当前选中时显示对勾

### 第一版不做

- 下拉面板内直接拖拽排序
- 分组折叠树
- 面板内复杂二级菜单

## 搜索栏

搜索框保留在顶部栏下方，紧接项目区之前。

### 行为规则

- 搜索作用域：当前分组下的项目列表
- 搜索维度：项目名 + 完整路径
- 匹配方式：大小写不敏感模糊匹配
- 为空时显示当前分组全部项目

占位文本可以保留类似参考图的视觉风格，但不实现 `(:active)` 语法解析。

## 项目列表

项目列表改为左侧面板的主体。

### 展示规则

- 以当前分组过滤后的项目为数据源
- 每个项目卡片显示：
  - 项目名
  - 缩略路径
  - 当前项目高亮态
  - 右侧操作区图标（保留“添加/设置”式视觉入口）

### 点击行为

- 点击项目卡片：调用现有 `openProject/switchProject` 切换项目
- 当前项目高亮，不重复切换

### 当前项目与分组的关系

- 当前项目不决定顶部显示内容；顶部显示当前分组
- 当前项目只决定项目卡片高亮

## 数据来源与职责边界

### 后端职责

后端继续只负责：

- 最近项目历史持久化
- 打开项目
- 关闭项目
- 返回最近项目列表

后端不负责：

- 项目分组
- 当前选中分组
- 分组排序
- 项目到分组的映射

### 前端职责

前端负责：

- 维护分组列表
- 维护项目到分组映射
- 维护当前选中分组
- 根据最近项目列表生成“分组后的项目列表”
- 搜索、筛选和顶部分组 UI

## 兼容旧数据策略

- 读取现有最近项目列表后，若某项目在 `projectGroupMap` 中没有映射，则自动归入 `未分组`
- `全部` 默认作为初始选中分组
- 原有 `ProjectSelector` 的“最近项目下拉”交互整体废弃

## 组件调整建议

### 需要新增/重构的前端单元

- `projectGroupingStore`
  - 负责分组、映射、当前分组、搜索状态
- `ProjectGroupHeader`
  - 顶部分组栏
- `ProjectGroupMenu`
  - 分组切换下拉面板
- `ProjectSearchBar`
  - 搜索栏
- `ProjectList`
  - 当前分组下的项目列表
- `ProjectListItem`
  - 项目卡片

### 现有组件关系

- `Sidebar.tsx` 负责左侧面板整体结构
- `ProjectSelector.tsx` 当前实现应被替换或拆分重构
- `projectStore.ts` 继续保留项目打开/最近项目能力

## 状态流

1. 应用启动时，`projectStore.loadRecentProjects()` 从后端加载最近项目
2. `projectGroupingStore` 读取本地持久化的分组和映射
3. 前端根据最近项目列表与映射生成当前分组项目列表
4. 用户切换分组时，更新 `selectedGroupId`
5. 用户搜索时，更新 `searchQuery`
6. 用户点击项目时，调用现有 `openProject`
7. 若用户打开了一个新项目，该项目自动出现在 `未分组` 中，除非已有映射

## 持久化方案

分组相关数据使用前端本地存储持久化：

- `groups`
- `projectGroupMap`
- `selectedGroupId`

搜索词 `searchQuery` 不持久化。

## 测试策略

### Store 测试

- 分组默认值
- 创建分组
- 重命名分组
- 删除分组后项目回落到 `未分组`
- 根据最近项目与映射生成过滤列表
- 搜索过滤逻辑

### 组件测试

- 顶部分组栏默认展示
- 点击下拉按钮展开分组面板
- 点击分组项切换当前分组
- `新建分组` 入口可见
- 当前分组项目列表渲染正确
- 点击项目调用现有切换链路

### 集成回归

- 左侧面板仍保持独立滚动
- 现有项目切换流程不被破坏
- 原有文件树 / Git 状态刷新链路保持可用

## 风险

- 当前左侧面板本来围绕 `Files / Changes / Worktrees` 设计，本次整改会把“项目导航中心”提升到更高优先级，组件结构需要重新拆分，不能只靠在 `ProjectSelector` 上继续堆功能。
- 分组状态放前端存储，意味着换机器不会同步；这是第一版的有意识取舍。
- 如果后续要把项目分组提升为核心能力，最终可能需要后端正式持久化结构，但本次不做这一步。
