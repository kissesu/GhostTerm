Tauri 前后端通信方案（修正版）

基于 PTY 二进制流特性重新分析，点击选择

核心洞察

GhostTerm 有两种通信模式，最优解可能不同：

请求/响应（文件、Git） 双向流（PTY）
频率 低频，用户触发 高频突发（编译输出数千行/秒）
数据类型 结构化 JSON 二进制字节流
最佳传输 Tauri Commands (天然适合) 需要评估
A

Commands + Events

优势

  • 零额外依赖，纯 Tauri 原生
  • 请求/响应操作完美适配
  • 架构简单，一种 IPC 通道

劣势

  • PTY 二进制数据需 base64 编码（+33% 体积）
  • 每条 Event 都经 JSON 序列化 + IPC bridge
  • 无背压机制，高频输出可能堆积
  • PTY 吞吐上限未验证

适合场景：PTY 输出量不大（日常 AI CLI 交互）

B

Commands + WebSocket（PTY 专用） ← 修正推荐

优势

  • WebSocket 原生二进制帧，零编码开销
  • xterm.js AttachAddon 直接对接，无需胶水代码
  • TCP 流控天然背压
  • 本地 WebSocket 吞吐远超 IPC bridge
  • 终端应用领域成熟方案（VS Code、Hyper 均用此模式）

劣势

  • 需管理本地 WebSocket server 生命周期
  • 端口分配 + 127.0.0.1 绑定 + token 验证
  • 两个通信通道增加架构复杂度

适合场景：高频终端数据流 + 结构化操作并存

C

Commands + Channel API

优势

  • Tauri 2 原生流式 API，专为此场景设计
  • 比 Events 开销低，可批量发送
  • 单一通信模式，架构最简洁

劣势

  • 仍需经过 IPC 序列化（二进制编码问题同 A）
  • 社区 PTY 场景实践案例较少
  • 性能上限不如原生 WebSocket

适合场景：追求架构简洁，愿意接受 IPC 瓶颈

修正后的推荐：B

PTY 用 WebSocket（二进制原生 + xterm.js 直连 + 背压控制），文件/Git 用 Tauri Commands（类型安全 + 零额外开销）。
这是"为不同问题选择不同最优解"，而非强行统一。
