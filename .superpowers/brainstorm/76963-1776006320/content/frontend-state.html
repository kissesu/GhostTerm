GhostTerm MVP -- 前端状态管理与数据流

Zustand 多 store 设计 + 模块间通信

Store 划分（3 个独立 store）

terminalStore

interface TerminalState {
  // PTY 连接信息
  ptyId: string | null
  wsPort: number | null
  wsToken: string | null
  connected: boolean

  // 操作
  spawn: (cwd: string) => Promise<void>
  kill: () => Promise<void>
  resize: (cols: number, rows: number)
    => Promise<void>
}

特点: 状态极少, 实际数据流走 WebSocket 不经过 store

editorStore

interface EditorState {
  // 打开的文件
  openFiles: OpenFile[]
  activeFileId: string | null

  // 操作
  openFile: (path: string) => Promise<void>
  closeFile: (id: string) => void
  saveFile: (id: string) => Promise<void>
  setActive: (id: string) => void

  // 外部文件变更处理
  handleExternalChange: (
    path: string,
    content: string
  ) => void
}

interface OpenFile {
  id: string        // 唯一标识
  path: string      // 文件路径
  content: string   // 当前编辑器内容
  diskContent: string // 磁盘内容(比对用)
  isDirty: boolean  // 是否有未保存修改
}

特点: 跟踪 dirty 状态, 处理 AI 修改文件后的冲突检测

sidebarStore

interface SidebarState {
  // 面板状态
  activeTab: 'files' | 'changes' | 'worktrees'
  visible: boolean

  // 项目
  currentProject: Project | null
  recentProjects: Project[]

  // 文件树
  fileTree: FileNode[]
  expandedDirs: Set<string>

  // Git
  changes: StatusEntry[]
  worktrees: Worktree[]

  // 操作
  setTab: (tab) => void
  toggleVisibility: () => void
  switchProject: (path) => Promise<void>
  refreshFileTree: () => Promise<void>
  refreshGitStatus: () => Promise<void>
  refreshWorktrees: () => Promise<void>
}

特点: 最复杂的 store, 管理文件树 + Git + 项目状态

数据流：文件打开到编辑到保存

用户点击文件树中的 auth.ts
1. FileTree.tsx  onClick(path)
     |
2. editorStore.openFile(path)
     |
3. invoke('read_file', { path })  →  Rust fs_backend
     |                                    |
4. 返回 { content, encoding }     ←------┘
     |
5. editorStore 更新:
     openFiles.push({ path, content, diskContent: content, isDirty: false })
     activeFileId = newId
     |
6. Editor.tsx 渲染:
     CodeMirror 加载 content
     根据文件扩展名选择语法高亮

数据流：AI 修改文件后的实时更新

Claude Code 在终端中修改了 auth.ts
1. Claude Code 写入磁盘 auth.ts
     |
2. notify watcher 检测到 Modify 事件 (Rust fs_backend)
     |
3. debounce 100ms 后读取新内容
     |
4. app.emit("fs:modified", { path, content })
     |
5. React listen("fs:modified") → editorStore.handleExternalChange(path, content)
     |
6. editorStore 判断:
     ├─ 文件未在编辑器中打开 → 忽略 (文件树会自动更新修改标记)
     ├─ 文件已打开, isDirty = false → 直接替换 content + diskContent
     └─ 文件已打开, isDirty = true → 弹出提示: "文件已被外部修改, 保留你的修改 / 加载新版本 / 查看 diff"

数据流：Worktree 切换

用户在 Worktrees 标签页点击 feat/auth
1. Worktrees.tsx  onClick(worktree)
     |
2. invoke('worktree_switch', { path: worktree.path })
     |
3. Rust project_manager 协调:
     ├─ fs_backend: stop_watching() → start_watching(new_path)
     ├─ pty_manager: 在当前 PTY 中 cd new_path (或 respawn)
     └─ 更新 projects.json
     |
4. 前端连锁更新:
     ├─ sidebarStore: currentProject 更新, 刷新 fileTree + changes
     ├─ editorStore: 关闭所有文件 (或提示保存未保存的)
     └─ terminalStore: cwd 已通过 Rust 端更新, 无需额外操作

模块间通信方式

场景 方式 理由
文件树 → 编辑器 (打开文件) editorStore.openFile() 直接调用目标 store 的 action, 最简单
Rust fs 事件 → 多个 store Tauri Event listener 分发 一个事件可能需要更新文件树 + 编辑器 + Git 状态
Worktree 切换 → 全局重置 Rust 端协调 + 前端 Event 响应 级联操作由 Rust project_manager 统一触发
编辑器保存 → Git 状态刷新 sidebarStore.refreshGitStatus() 保存后主动刷新, 比依赖 fs 事件更及时
