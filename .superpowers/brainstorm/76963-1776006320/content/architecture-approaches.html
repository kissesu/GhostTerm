GhostTerm MVP -- 前端架构方案

三种不同的 React 组件组织和状态管理方式，点击选择你倾向的方案

A

Feature-Sliced (推荐)

思路：按功能切片组织，每个功能（终端、编辑器、文件树）是独立模块，包含自己的组件、hooks、store

状态管理：Zustand — 每个功能一个独立 store，通过 subscription 跨模块通信

优势：模块隔离清晰，新增功能只加目录不改现有代码，适合渐进式开发

劣势：初期目录结构略多

src/
  features/
    terminal/       # 终端模块
      Terminal.tsx
      useTerminal.ts
      terminalStore.ts
    editor/         # 编辑器模块
      Editor.tsx
      useEditor.ts
      editorStore.ts
    sidebar/        # 左侧面板
      Sidebar.tsx
      FileTree.tsx
      Changes.tsx
      Worktrees.tsx
      sidebarStore.ts
  shared/           # 共享组件/工具
    components/
    hooks/
    types/
  layouts/
    AppLayout.tsx   # 分屏布局容器
B

Flat + Context

思路：扁平目录，components/hooks/stores 平级分开，用 React Context 管理全局状态

状态管理：React Context + useReducer — 内置方案，零依赖

优势：最简单，无额外依赖，小项目足够

劣势：Context 更新会导致大范围重渲染；文件增多后难以定位功能边界

src/
  components/
    Terminal.tsx
    Editor.tsx
    Sidebar.tsx
    FileTree.tsx
    ...
  hooks/
    useTerminal.ts
    useEditor.ts
    ...
  contexts/
    AppContext.tsx
    TerminalContext.tsx
  layouts/
    AppLayout.tsx
C

Minimal Monolith

思路：MVP 极简 — 尽量少的文件，快速出原型，等功能稳定后再重构

状态管理：Zustand 单 store — 一个 store 管所有状态

优势：最快能跑起来，初期开发速度最快

劣势：Supremum 的教训（4849行上帝组件）就是这条路走到底的结果

src/
  App.tsx
  Terminal.tsx
  Editor.tsx
  Sidebar.tsx
  store.ts        # 单一 store
  types.ts

关键对比

A. Feature-Sliced B. Flat + Context C. Minimal
初期速度 中 中 快
Phase 2 扩展性 好 中 差
性能 好（精准更新） Context 重渲染 好（Zustand）
额外依赖 zustand 无 zustand
避免 Supremum 问题 是 部分 否
