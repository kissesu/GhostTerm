Tauri 前后端通信方案

React 前端如何与 Rust 后端通信（PTY 控制、文件系统、Git 操作）

A

Tauri Commands + Events (推荐)

思路：Tauri 原生方案。用 #[tauri::command] 做请求/响应，用 Events 做后端推送（如终端输出流）

通信模型：

  • 前端 → 后端：invoke('write_pty', { data }) 命令调用
  • 后端 → 前端：app.emit('pty-output', data) 事件推送
  • 文件/Git 操作：同步 command 即可

优势：零额外依赖，类型安全（tauri-specta 可自动生成 TS 类型），Tauri 2 官方推荐

劣势：大量二进制数据传输时需注意序列化开销

B

Tauri Commands + WebSocket（PTY 专用）

思路：文件/Git 操作用 Tauri Commands，但 PTY 数据流走本地 WebSocket

通信模型：

  • 文件/Git：invoke('read_file', { path })
  • PTY 数据流：WebSocket ws://localhost:PORT

优势：WebSocket 对高频双向流更高效，xterm.js 原生支持 WebSocket attach

劣势：多一个通信通道，需管理端口和连接生命周期

C

纯 Tauri Commands（全部统一）

思路：所有通信都走 Tauri Commands，PTY 输出通过轮询或 Channel API

通信模型：

  • 所有操作：invoke('xxx', { ... })
  • PTY 输出：Tauri 2 Channel API（streaming response）

优势：最简单，单一通信模式

劣势：Channel API 相对较新，PTY 高频数据场景验证较少

数据流示意（方案 A）

┌─────────────────────────────────────────────────┐
│                React Frontend                    │
│                                                  │
│  Terminal.tsx ──invoke('write_pty')──┐            │
│       ↑                              ↓            │
│  listen('pty-output')        Tauri IPC Bridge    │
│       ↑                              ↓            │
│  Editor.tsx ──invoke('read_file')───┘            │
│  Sidebar.tsx ──invoke('list_dir')───┘            │
└─────────────────────────────────────────────────┘
                       ↕
┌─────────────────────────────────────────────────┐
│                 Rust Backend                      │
│                                                  │
│  pty_manager.rs    (spawn/write/resize PTY)      │
│  fs_backend.rs     (read/write/list files)       │
│  git_backend.rs    (status/stage/worktree)       │
│                                                  │
│  PTY 输出 → app.emit('pty-output', bytes)        │
└─────────────────────────────────────────────────┘
